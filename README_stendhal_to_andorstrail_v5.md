# stendhal_to_andorstrail_v5.py — User Guide

## Overview

`stendhal_to_andorstrail_v5.py` converts the entire Stendhal MMORPG map set into
Andor's Trail-compatible 32×32-tile TMX map chunks, including every overworld zone,
underground dungeon, and interior building on all levels.

It is a zero-dependency Python 3.6+ script. No pip packages are required.

---

## What it does

| Step | Description |
|------|-------------|
| 1 | Reads every zone definition from `data/conf/zones/*.xml` |
| 2 | Loads the corresponding Tiled TMX map files from `data/maps/` |
| 3 | Parses all portal/staircase/entrance XML elements to extract exit links |
| 4 | Splits each Stendhal zone into 32×32-tile AT chunks |
| 5 | Writes one AT-compatible TMX file per chunk under `res/xml/` |
| 6 | Writes `worldmap.xml`, `loadresources.xml`, and a Tiled `.world` file |
| 7 | Writes `res/xml/portal_exits_catalog.txt` — a sorted list of every exit |

---

## Version history

| Version | Key change |
|---------|-----------|
| v3 | Interior zones (`int_*`) get `INT_LEVEL=1000`; ref-based portal naming |
| v4 | All interior map exits emitted; two-pass XML ref resolution |
| **v5.0** | **Bidirectional exits** — both sides of every portal get TMX objects; **typed mapchange** objects (`mapchange_up`, `mapchange_down`, `mapchange_enter`, `mapchange_exit`); **portal catalog file** |
| **v5.1** | **Fixed zero-exit bug**: Pass 1 now uses recursive `iter()` (was direct children only); handles namespaced `<destination>` tags; supports direct x/y destination coords; Pass 3 uses wider property name patterns and Tiled `class` attribute; diagnostic `[PROBE]` output when zero portals found; `portals_override.json` fallback |

## Requirements

- **Python 3.6 or newer** (standard library only — no pip install needed)
- The **Stendhal project source tree** checked out locally
  (the directory that contains `data/`, `src/`, etc.)

## How to run

```bash
# 1. Clone or download the Stendhal project
git clone https://github.com/arianne/stendhal.git
cd stendhal

# 2. Place this script anywhere (or run from outside)
python3 /path/to/stendhal_to_andorstrail_v5.py
```

The script **must be run from the Stendhal project root** — the directory that
contains `data/conf/zones` and `data/maps`.  Output is written into a `res/`
sub-directory created automatically.

### Output directories

```
res/
├── xml/                        ← AT TMX map chunks  +  worldmap.xml
│   ├── 0_semos_city_x0_y0.tmx
│   ├── 0_semos_city_x1_y0.tmx
│   ├── ...
│   ├── worldmap.xml
│   ├── portal_exits_catalog.txt   ← NEW in v5
│   └── ...
├── drawable/                   ← copied tileset PNG images
└── values/
    └── loadresources.xml       ← Android resource array of all map IDs
stendhal_world.world            ← Tiled world file
```

## Exit types — what's new in v5

Every portal, staircase, and building entrance now writes **two** TMX
`<object>` elements: one in the source chunk and one in the destination chunk.
Both objects get a typed `type=` attribute:

| Type | Meaning | Example |
|------|---------|---------|
| `mapchange_up` | Ascending to a higher numeric level | Staircase from level -1 to level 0 (dungeon to surface) |
| `mapchange_down` | Descending to a lower numeric level | Staircase from level 0 to level -1 (entering a dungeon) |
| `mapchange_enter` | Entering an interior zone (`int_*`) | Walking into a shop, house, or guild hall |
| `mapchange_exit` | Leaving an interior zone | Walking out of a building back to the overworld |
| `mapchange` | Same-level horizontal zone bridge | Moving between adjacent outdoor zones on level 0 |

In v4 every exit used `type="mapchange"` regardless of direction.  In v5
this distinction lets Andor's Trail apply different animations or transitions
for going upstairs vs. downstairs vs. entering a building.

### How exits look in a TMX file

```xml
<!-- Staircase going DOWN from level 0 to level -1 -->
<object name="to_dungeon_entrance" type="mapchange_down"
        id="3" x="256" y="320" width="32" height="32">
  <properties>
    <property name="map"   value="neg1_semos_dungeon_x2_y1"/>
    <property name="place" value="from_surface"/>
  </properties>
</object>

<!-- Reciprocal exit in the dungeon chunk (mapchange_up back to surface) -->
<object name="from_surface" type="mapchange_up"
        id="1" x="64" y="32" width="32" height="32">
  <properties>
    <property name="map"   value="0_semos_mountain_n2_x0_y0"/>
    <property name="place" value="to_dungeon_entrance"/>
  </properties>
</object>
```

## The portal exits catalog

After a successful run, `res/xml/portal_exits_catalog.txt` lists every portal
exit in the entire world.  Exits are sorted by type: up/down/enter/exit first,
then horizontal.

```
portal_exits_catalog.txt
Generated by stendhal_to_andorstrail_v5.py
============================================...
SOURCE CHUNK                                    TARGET CHUNK                     [TYPE]                NAME / PLACE
--------------------------------------------...
0_semos_mountain_n2_x0_y0               → neg1_semos_dungeon_x2_y1        [mapchange_down   ]  name=to_dungeon_entrance  place=from_surface
neg1_semos_dungeon_x2_y1                → 0_semos_mountain_n2_x0_y0       [mapchange_up     ]  name=from_surface  place=to_dungeon_entrance
0_semos_city_x0_y0                      → int_semos_bank_x0_y0             [mapchange_enter  ]  name=bank_entrance  place=exit
int_semos_bank_x0_y0                    → 0_semos_city_x0_y0               [mapchange_exit   ]  name=exit  place=bank_entrance
...
```

This file makes it straightforward to:
- Find every map that has an upstairs or downstairs exit
- Verify interior building entrances are correctly paired
- Debug missing or unresolved portal links

## Maps with up / down exits (by zone name pattern)

The following Stendhal zone name patterns indicate zones that connect to a
different numeric level and will therefore produce `mapchange_up` /
`mapchange_down` exits in their TMX chunks.  After running the script, check
`portal_exits_catalog.txt` for the exact chunk IDs.

### Zones that descend (enter dungeons / underground)
| Zone name pattern | Where it goes |
|-------------------|---------------|
| `0_semos_mountain*` | Connects to `-1_*` dungeon entrance zones |
| `0_ados_rock*` | Connects to `-1_ados_dungeon*` |
| `0_nalwor_city` | Connects to `-1_nalwor_forest*` (underground passages) |
| `0_fado_*` | Connects to `-1_fado_forest*` |
| `0_deniran_*` | Connects to `-1_deniran*` under-city |
| Any `0_*` zone with a staircase portal | May connect to `-1_*` |
| Any `-1_*` zone with a staircase portal | May connect to `-2_*` |
| Any `-N_*` zone | May connect to `-(N+1)_*` (multi-level dungeons) |

### Zones that ascend (surface exits from dungeons)
All of the `-1_*`, `-2_*`, `-3_*`, `-4_*` level zones that have a staircase
portal pointing to a higher level produce `mapchange_up` exits.

### Interior zones (enter / exit)
Every zone whose name starts with `int_` (or whose XML `level` attribute is
`"int"`) is classified as an interior zone.  They produce `mapchange_enter`
exits in the overworld chunks and `mapchange_exit` exits in the interior chunk.

Common interior zone name patterns:

| Pattern | Location in Stendhal |
|---------|----------------------|
| `int_semos_*` | Buildings in Semos city |
| `int_ados_*` | Buildings in Ados |
| `int_kirdneh_*` | Buildings in Kirdneh |
| `int_fado_*` | Buildings in Fado |
| `int_nalwor_*` | Buildings in Nalwor |
| `int_deniran_*` | Buildings in Deniran |
| `int_afterlife` | The afterlife realm |
| `int_hell_*` | Hell maps |
| `int_amazon_*` | Amazon island buildings |
| `int_*_dungeon_*` | Interior dungeon rooms |
| `int_*_guild*` | Guild halls |

## How to modify the script

### Change output directories

At the top of the file, edit the path constants:

```python
STENDHAL_ZONE_CONF = "data/conf/zones"    # where Stendhal zone XML files live
DATA_MAPS_BASE     = "data/maps"          # where Stendhal TMX map files live
OUTPUT_XML         = os.path.join("res", "xml")        # AT TMX output
OUTPUT_DRAWABLE    = os.path.join("res", "drawable")   # copied tilesets
OUTPUT_VALUES      = os.path.join("res", "values")     # loadresources.xml
WORLD_FILE         = "stendhal_world.world"            # Tiled world file
CATALOG_FILE       = "portal_exits_catalog.txt"        # exit catalog filename
```

### Change chunk size

AT uses 32×32 tiles.  If you need a different size:

```python
AT_W    = 32    # chunk width  in tiles
AT_H    = 32    # chunk height in tiles
TILE_PX = 32    # pixels per tile
```

Changing `AT_W`/`AT_H` will alter how many TMX files are produced and re-tile
all exit positions accordingly.

### Change which zones are treated as "interior"

The `INT_LEVEL = 1000` sentinel identifies interior zones.  Two rules determine
whether a zone is interior:

1. Its name starts with `int_`
2. Its XML `level` attribute is one of `"int"`, `"interior"`, or `"i"`

To add extra interior zone patterns, edit `load_zone_configs`:

```python
# Example: treat zones starting with "inn_" as interior too
if is_interior_level(raw_level) or name.startswith("int_") or name.startswith("inn_"):
    level = INT_LEVEL
```

### Change exit type names

Exit type strings are constants at the top of the file.  Change them here and
the new names will propagate everywhere:

```python
EXIT_UP     = "mapchange_up"
EXIT_DOWN   = "mapchange_down"
EXIT_ENTER  = "mapchange_enter"
EXIT_EXIT   = "mapchange_exit"
EXIT_HORIZ  = "mapchange"
```

For example, to use Andor's Trail's numeric warp type if it ever gains one:

```python
EXIT_UP    = "warp_ascend"
EXIT_DOWN  = "warp_descend"
```

### Add extra portal XML tags to scan

All recognised portal element names are in the `PORTAL_TAGS` constant at the
top of the file.  Add your custom tag name there:

```python
PORTAL_TAGS = frozenset((
    "portal", "teleporter", "teleport", "entrance", "staircase",
    "stair", "exit", "transition", "warp", "door", "link",
    "my_custom_tag",    # ← add here
))
```

### Add extra TMX object type keywords for the fallback scan

Pass 3 uses a regex (`PORTAL_TYPE_RE`) to identify portal-like objects in TMX
objectgroups.  Extend the alternation pattern:

```python
PORTAL_TYPE_RE = re.compile(
    r"portal|teleport|staircase|stair|entrance|exit|transition|warp"
    r"|door|link|zone.?change|mapchange|my_keyword",   # ← add here
    re.IGNORECASE,
)
```

### Add extra destination zone property names

Pass 3 checks a list of property name patterns for a destination zone.  Add
yours to `DEST_ZONE_PROPS`:

```python
DEST_ZONE_PROPS = (
    "destination-zone", ...,
    "my_custom_dest_prop",    # ← add here
)
```

### Suppress one side of bidirectional exits

If you only want the forward exit (v4 behaviour), comment out the reverse-exit
block in `build_portal_exits`:

```python
# ── Reverse exit: tgt_chunk → src_chunk ──────────────────────────────
# COMMENT THIS ENTIRE BLOCK to revert to v4 single-direction behaviour
if rev_name not in used_names[tgt_chunk]:
    ...
```

### Use only XML-resolved portals (skip TMX fallback scan)

To skip Pass 3 (TMX objectgroup fallback) and rely only on zone config XML:

```python
# In load_zone_portals, comment out or delete the Pass 3 block:
# ── Pass 3: TMX objectgroup fallback ──────────────────────────────────
# for zone in zones:
#     ...
tmx_fallback_count = 0
return portals, raw_portals, tmx_fallback_count
```

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| `[ERROR] Cannot find 'data/conf/zones'` | Not run from Stendhal root | `cd /path/to/stendhal` then re-run |
| Many `[WARN] Not found: …tmx` lines | TMX files in unexpected sub-dirs | Check `DATA_MAPS_BASE`; may need to point to a specific level subdirectory |
| `[WARN] Tileset image not found` | Tileset PNG not in expected location | Copy tilesets to `res/drawable/` manually or adjust `_resolve_image` |
| Portal exits = 0 (with `[PROBE]` output) | Zone XML has no portal child elements | Read the probe output — portals may be Java-only; use `portals_override.json` (see below) |
| Portal exits = 0, probe shows tags | Tag names differ from expected | Add the tag name shown in the probe to `PORTAL_TAGS` at the top of the script |
| Pass 3 found 0 from TMX | TMX objectgroups have no portal objects | Normal for Stendhal — portals are in Java code; use `portals_override.json` |
| All exits typed `mapchange` only | Level attributes missing from zone XML | Verify zone XML has numeric or `"int"` level attributes |
| Duplicate exit names in one chunk | Two portals with the same ref on the same tile | Stendhal data issue; the second portal is silently skipped (deduplication) |

### What to do when portal exits stay at 0

Run the script once — if portals are 0 you will see a `[PROBE]` section in
the console output.  Read the probe to understand why:

**Case A — probe says "No child elements found inside `<zone>` elements":**
Your Stendhal version defines portals only in Java source code.  They are not
in any XML or TMX file the script can read.  Use `portals_override.json` (see
below) to supply them manually.

**Case B — probe shows child tags but not portal tags:**
The XML uses a different element name.  Add that name to `PORTAL_TAGS` at the
top of the script.

**Case C — probe shows portal tags, but exits are still 0:**
The destination format is probably different.  Paste one of the sample portal
elements from the probe output here and we can pinpoint the fix.

### portals_override.json — manual portal data

Create a file named `portals_override.json` in the Stendhal project root.  The
script picks it up automatically on every run.

```json
[
  {
    "src_zone": "0_semos_city",
    "src_x": 19,
    "src_y": 4,
    "tgt_zone": "int_semos_bank",
    "tgt_x": 2,
    "tgt_y": 3,
    "src_ref": "bank_entrance",
    "tgt_ref": "exit"
  },
  {
    "src_zone": "0_semos_mountain_n2",
    "src_x": 55,
    "src_y": 110,
    "tgt_zone": "-1_semos_dungeon",
    "tgt_x": 3,
    "tgt_y": 3
  }
]
```

Fields:

| Field | Required | Description |
|-------|----------|-------------|
| `src_zone` | ✓ | Stendhal zone name of the source side |
| `src_x`, `src_y` | ✓ | Tile coordinates of the portal tile in the source zone |
| `tgt_zone` | ✓ | Stendhal zone name of the destination |
| `tgt_x`, `tgt_y` | optional | Tile coordinates in the destination zone (for the reverse exit) |
| `src_ref` | optional | AT exit `name` string (used in the source chunk TMX) |
| `tgt_ref` | optional | AT exit `name` string (used in the destination chunk TMX) |

If `tgt_x`/`tgt_y` are omitted, only the forward exit is written (no reverse).
`src_ref`/`tgt_ref` control what the exit objects are named; if omitted the
script generates names automatically (`enter`, `exit`, `up`, `down`, etc.).
